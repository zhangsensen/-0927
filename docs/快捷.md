 1. 调研阶段（RESEARCH）
     - 触发：在编辑器输入 `rresearch` 然后按 Tab
     - 展开示例：
       ```text
       ENTER RESEARCH MODE

       现在先帮我进入调研阶段：只阅读相关文件，总结现有实现和可能的风险点，不要给任何解决方案或代码。
       看完后用中文总结你理解的需求和代码结构，如果有不清楚的地方先提问。
       ```

  2. 方案阶段（INNOVATE）
     - 触发：输入 `rinnovate` 然后按 Tab
     - 展开示例：
       ```text
       ENTER INNOVATE MODE

       基于刚才的分析，现在进入方案阶段：请用中文给出 2–3 个可行思路，简单比较优缺点（复杂度、可维护性、风险）。
       先不要写代码，也不要做最终决策，等我选。
       ```

  3. 计划阶段（PLAN）
     - 触发：输入 `rplan` 然后按 Tab
     - 展开示例：
       ```text
       ENTER PLAN MODE

       按我选的这个方案进入计划阶段：列出一个实现计划，包含要改动的文件路径、函数名/模块名、主要改动点、可能的边界
  情况。
       以及一个 IMPLEMENTATION CHECKLIST（每一步是原子操作）。先把计划给我确认，不要写代码。
       ```

  4. 实现阶段（EXECUTE）
     - 触发：输入 `rexecute` 然后按 Tab
     - 展开示例：
       ```text
       ENTER EXECUTE MODE

       现在按刚才确认的 IMPLEMENTATION CHECKLIST 进入实现阶段：严格按步骤实现，不要擅自增加新步骤或重构无关代码。
       每完成几步，说明你完成了 checklist 的哪些项，并只给出与这些步骤相关的代码改动。
       ```

  5. 自查阶段（REVIEW）
     - 触发：输入 `rreview` 然后按 Tab
     - 展开示例：
       ```text
       ENTER REVIEW MODE

       现在进入自查阶段：对照我们之前的 PLAN 和 checklist，用中文总结这次改动的内容，并逐条确认每一项是否按计划
  完成。
       如果有任何偏离，请用“DEVIATION DETECTED: …” 明确标出，并给出建议的修正方案，同时列出我需要重点测试的用例。
       ```

  6. 小任务简化流程
     - 触发：输入 `rsmall` 然后按 Tab
     - 用途：小改动时一条指令压缩“理解→方案→实现→自查”
     - 展开示例：
       ```text
       这次是一个小改动，但你仍然按“先理解→再简短给方案→再写代码→最后简单自查”的顺序来回答，不用显式切换模式。
       先用 2–3 句中文确认你理解的需求，然后再给出修改建议和代码。



         这是一个较大的开发/研究项目，请你在整个对话里**自动**按以下 5 个阶段推进：RESEARCH → INNOVATE → PLAN → EXECUTE
  → REVIEW。

  全程规则：
  - 你自己负责推进阶段，不需要我每次手动切模式。
  - 每进入下一个阶段前，用 1–2 句话说明你要做什么，并等待我一句话确认后再继续。
  - 任何时候如果信息不足，先提问澄清，不要自行假设关键需求。

  具体阶段要求：
  1）RESEARCH（调研阶段）
  - 只阅读我给的需求和相关文件（如果有），总结现有实现、约束和风险点。
  - 不要提出任何方案或写代码，只做理解和归纳。
  - 输出：用中文总结你理解的需求、现有结构和疑点列表。

  2）INNOVATE（方案阶段）
  - 基于 RESEARCH，总结 2–3 个可行思路。
  - 简要比较每个思路的优缺点（复杂度、可维护性、风险）。
  - 不要写代码，也不要最终拍板，由我来选或让你综合。

  3）PLAN（计划阶段）
  - 按我选定或你综合出的方案，给出实现计划。
  - 包含：要改动的文件路径、模块/函数名、主要改动点、可能的边界情况。
  - 输出一个「IMPLEMENTATION CHECKLIST」，每一步是可以单独完成和检查的原子操作。

  4）EXECUTE（实现阶段）
  - 严格按照 checklist 的顺序实现，不要无关重构。
  - 每完成若干步，说明你完成了 checklist 的哪些项，并只展示相关代码改动。
  - 遇到不确定的地方暂停，向我确认后再继续。

  5）REVIEW（自查阶段）
  - 对照 PLAN 和 checklist，总结本次改动内容。
  - 逐条确认每一项是否完成。
  - 对于任何偏离，用“DEVIATION DETECTED: …” 标出，并给出修正建议。
  - 列出我需要重点测试的用例或场景。
